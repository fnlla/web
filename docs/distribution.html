Choose the handoff

Distribution is mostly a decision about what another project should copy. The safe default is the generated runtime export, while the raw repository runtime files are mainly for framework maintainers working in-place.

Default: copy dist/fnlla-ui/

Use the generated export when another project or another team needs FNLLA UI as a vendored dependency. It contains the runtime tree plus the minimal release files needed for version and license tracking.

dist/
  fnlla-ui/
    assets/
      css/
        fnlla-ui.css
      js/
        fnlla-ui.js
      icons/
    LICENSE.md
    README.md
    VERSION

Use raw assets/ only when

  • You are working directly inside the FNLLA UI repository.
  • You need the live published runtime next to src/, scripts and docs while maintaining the framework.
  • You are validating that publish output, docs and browser behavior all describe the same current snapshot.

Runtime boundary

The supported runtime API is intentionally narrow. If a consumer page only depends on FNLLA UI, these are the files it should know about.

What belongs to runtime

  • assets/css/fnlla-ui.css is the published stylesheet that pages should link.
  • assets/js/fnlla-ui.js is the published script that pages should load for documented behaviors.
  • assets/icons/ holds the offline icon bundle, including root SVG files, sprite.svg, notice and license files.
  • dist/fnlla-ui/ is the clean handoff form of the same runtime contract.
<link rel="stylesheet" href="fnlla-ui/assets/css/fnlla-ui.css">
<script src="fnlla-ui/assets/js/fnlla-ui.js"></script>

What stays outside runtime

  • Do not link src/css/... or src/js/.... Those are maintainer source files, not consumer dependencies.
  • Do not link docs/assets/docs.css or docs/assets/docs.js. They style and enhance the documentation product only.
  • Do not point production pages at docs-relative paths. Consumer pages should stay on runtime paths such as assets/css/fnlla-ui.css and assets/icons/search.svg.

Repository zones

FNLLA UI has a source-to-runtime pipeline. Each zone has a distinct job and should stay on that side of the boundary.

Published runtime

assets/css/, assets/js/ and assets/icons/ are the stable browser-facing product surface inside the repository.

Runtime export

dist/fnlla-ui/ is the downstream handoff package. It is generated, not hand-maintained.

Maintainer source

src/css/ and src/js/ are where maintainers change tokens, components and behavior modules before republishing readable runtime files.

Documentation product

docs/ proves and explains the framework contract, but docs/assets/ remains documentation-only support code.

Release discipline

Packaging stays trustworthy only when source, published runtime, generated docs and validation all move together.

Maintainer flow

  1. Edit source modules

    Change tokens, components or interaction code under src/ first.

  2. Republish runtime and docs

    Run node .\scripts\publish-fnlla-ui.mjs so runtime files, generated guides and dist/fnlla-ui/ all match source.

  3. Smoke-test browser behavior

    Run node .\scripts\test-fnlla-ui-browser.mjs when runtime behavior or packaging changed.

  4. Validate the repository contract

    Run node .\scripts\validate-fnlla-ui.mjs to confirm docs shell sync, release files and generated outputs.

Downstream project structure

my-site/
  fnlla-ui/
    assets/
      css/
        fnlla-ui.css
      js/
        fnlla-ui.js
      icons/
    LICENSE.md
    README.md
    VERSION
  assets/
    css/
      site.css
    js/
      site.js
  index.html
  about.html
  contact.html

This keeps FNLLA UI as one vendored runtime layer while project-specific CSS, JS and templates stay in local project files instead of drifting back into the framework package.
