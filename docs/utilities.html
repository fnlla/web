Scrollbar utilities

Apply .scrollbar to any scroll container and add .scrollbar-thin when the chrome should stay visually lighter.

Scrollable panel

The scrollbar reads from shared theme tokens, so the same utility stays usable in default and dark theme contexts.

Use it for sidebars, long dropdown groups, dense inspector panels, code-like wrappers or documentation navigation.

The framework keeps scrollbar styling opt-in, which avoids forcing a custom chrome onto every browser surface by default.

Pair it with .overflow-auto on containers that can exceed their available height or width.

<div class="overflow-auto scrollbar scrollbar-thin">
  ...
</div>

When utilities are enough

Utilities are strongest when they finish a layout or presentation detail. They should not quietly replace the documented page shell or component roles.

Good fit

  • Adjusting local spacing inside an already-correct layout structure.
  • Aligning actions, badges or support text inside an existing card or toolbar.
  • Making a preview, table or sidebar scroll safely without inventing new wrappers.
  • Applying one-off muted text, width or radius tweaks that still follow shared tokens.

Step back to layout or components

  • If the page shell itself is unclear, start from layout.html before stacking utilities.
  • If a repeated pattern needs semantic structure, prefer a documented component or section instead of a growing utility pile.
  • If a utility combination is copied across pages, it is usually time to promote that pattern into a reusable surface.
  • If visual state depends on JavaScript behavior, verify the supported runtime hook in contract.html rather than styling around it ad hoc.

Compact composition example

This is the sweet spot for utilities: a small, readable composition inside a stable card and section structure.

<div class="card p-3 radius-md shadow-sm">

      <p class="text-muted mb-0">Production-ready handoff</p>
    </div>
    <span class="tag">Ready</span>
  </div>

Theme-aware token usage

The utility layer stays theme-aware because color, shadow and surface values all read from shared tokens.

Theme attribute examples

<body data-fnlla-theme="default">
<body data-fnlla-theme="dark">

Theme switching works by overriding CSS variables only, which keeps the utility classes reusable.

Token groups available

Colors: semantic surface, text, border, overlay and accent tokens.

Typography: font families, scale, line heights and weights.

Layout: spacing, sizing, containers, breakpoints, z-index and transitions.

Local icon bundle and offline catalogue

FNLLA Web ships a local assets/icons/ bundle as part of the runtime contract. Use it when a page needs durable offline icon paths, predictable sprite usage and a searchable catalogue without any external request.

Bundle contract

  • All previewed assets come from the local FNLLA Icons bundle.
  • Copy buttons return the stable runtime path under assets/icons/, not a docs-relative path.
  • Aliases are merged so the catalogue shows each shipped SVG once, even when multiple icon names resolve to the same file.
  • Keep visible text for meaning. Icons should reinforce labels, not replace them.

Runtime file path

<span class="icon" aria-hidden="true">
  <img src="assets/icons/search.svg" alt="">
</span>

If a downstream project vendors the framework under a folder such as fnlla-web/, prepend that folder in the project markup, for example fnlla-web/assets/icons/search.svg.

Local sprite usage

<span class="icon" aria-hidden="true">
  <svg><use href="assets/icons/sprite.svg#settings-2"></use></svg>
</span>

Use individual SVG files when you want the clearest offline file reference. Use the local sprite when a page reuses many icons and the bundle path is already in place.
