Form patterns

These examples show labels, inputs, textarea fields, select elements, choice controls, range and file inputs, validation states, status messages and reusable layout recipes for production-ready form flows. Single-select fields are runtime-enhanced so the closed control and open option list stay visually consistent across supported browsers.

Layer Stable form contract Authoring rule
Structure .form, .form-group, .form-row, .fieldset Start from predictable grouping before styling special cases. Layout decisions should stay readable in raw markup.
Controls .input, .select, .textarea, .range-input, .file-field, choice controls and switches Prefer the shipped field shells before inventing local variants for common input types.
Supporting text .help-text, .success-text, .error-text Connect contextual and validation copy with aria-describedby so support text remains part of the field contract.
Submission state .form-message, .form-message-success, .form-message-error Use one visible summary near the top for form-level outcome, then repeat specific issues or confirmations at field level when needed.
Compositions .input-group, .contact-section, compact capture and long-form stacked layouts Use composition patterns when the form needs a reusable shell, not just a list of standalone controls.

Authoring guidance

These notes help decide which layout to use and how to keep the submission flow understandable.

Choosing a layout

  • Use one-column stacking for long answers, review-heavy forms or anything likely to be completed on mobile.
  • Use two-column grids only for short related fields such as name and company or email and phone.
  • Use compact capture forms when the page goal is one quick next step rather than a full qualification flow.

Submission flow checklist

  • Before submit, make sure every field has a visible label, useful autocomplete hint when relevant, and concise support text only where needed.
  • After a failed submit, keep one visible summary near the top and repeat the specific message at each invalid field.
  • After a successful submit, keep the confirmation visible long enough for the user to understand the next step.

Accessibility notes

These form notes help keep the framework examples usable when adapted into real products.

Labels and semantics

  • Every input, select and textarea should have a visible label.
  • Placeholders can support the field, but should not replace a label.
  • Use proper input types and autocomplete hints where they improve usability.
  • In input groups, visible prefixes and suffixes should supplement the label, not replace it.

Help and error text

  • Link every relevant help, success or error message to the control with aria-describedby.
  • Use aria-invalid="true" when a field is actually in an error state.
  • Keep help and error messages close to the relevant field.

Keyboard and mobile use

  • Fields must remain fully visible and tappable on smaller screens.
  • Buttons should stay large enough to tap without crowding the layout.
  • Long forms should use stacking layouts instead of forcing horizontal scrolling.

Validation

  • Pair native validation with clear field-level messaging and, when needed, one form-level summary near the top.
  • Do not rely only on color to indicate an error state.
  • Use role="status" for polite success confirmation and role="alert" only when failed submission feedback should be announced immediately.
  • In a real application flow, move focus to the summary or first invalid field after failed submission.
  • Always keep the submission flow usable with keyboard only.
