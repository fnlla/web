Component contracts

These rules define the intended markup contract for the interactive patterns shipped with the framework.

Navigation

  • Use a real toggle button with data-fnlla-nav-toggle and aria-controls pointing to the collapsible panel.
  • Wrap the menu list and CTA area inside one .navbar-panel when you want a unified mobile panel.
  • Keep the primary links in .navbar-menu inside the shared mobile panel.
  • Escape closes the mobile panel and returns focus to the toggle so keyboard users do not stay inside hidden navigation content.

Alerts

  • Use .alert as the base wrapper and add exactly one semantic variant such as .alert-info or .alert-danger.
  • Use .alert-title and .alert-text for a predictable heading and body structure.
  • Use role="status" for passive updates and reserve role="alert" for urgent failures or warnings.

List groups and empty states

  • Use .list-group when multiple related rows should share one surface instead of repeated isolated cards.
  • Add .list-group-item-action only when the whole row should feel interactive.
  • Use .empty-state whenever a panel would otherwise be visually blank and the page should suggest a next step.

Dropdown

  • Use [data-fnlla-dropdown] on the wrapper and [data-fnlla-dropdown-toggle] on a real button.
  • The toggle should control one .dropdown-menu through aria-controls.
  • Keep focusable actions inside the menu so arrow-key movement stays predictable.
  • Arrow Down, Arrow Up, Home and End are supported while the dropdown is open.

Tabs

  • Use [data-fnlla-tabs] on the wrapper, [data-fnlla-tab-list] on the tab row and [data-fnlla-tab] on each real button.
  • Each tab button should point to one .tab-panel through aria-controls, and each panel should point back with aria-labelledby.
  • Mark one initial tab with aria-selected="true"; the script will keep the rest synchronized.
  • Horizontal tablists use Arrow Left and Arrow Right; vertical tablists should declare aria-orientation="vertical" and use Arrow Up and Arrow Down instead.
  • Home and End move to the first and last tab and activate the newly focused panel.

Breadcrumbs

  • Wrap breadcrumbs in a labeled nav and keep the trail inside an ordered list.
  • Use .breadcrumb-link for earlier steps and one .breadcrumb-current item with aria-current="page" for the current location.
  • Keep labels short enough to wrap cleanly without breaking the reading order.

Pagination

  • Wrap pagination in a labeled nav and keep page actions in a list so the sequence stays understandable to assistive technology.
  • Use aria-current="page" on the active page link.
  • Use visible Previous and Next labels when they improve clarity on smaller screens.

Tables

  • Wrap wider tables in .table-responsive so the layout degrades to horizontal scrolling instead of breaking the page width.
  • Keep table headers as real th elements and reserve striped or hover states for cases where scanning row relationships matters.
  • Prefer tables for dense comparative data, not for general card-like marketing content.

Modal

  • Use [data-fnlla-modal-open] on the trigger and point it at one modal selector such as #demo-modal.
  • Mark the overlay with [data-fnlla-modal] and keep one .modal-content dialog container inside it.
  • Put dialog semantics and labeling on the modal container with role="dialog", aria-modal, aria-labelledby and, when needed, aria-describedby.
  • Always provide a visible close control with [data-fnlla-modal-close].
  • Use data-fnlla-modal-initial-focus or native autofocus when one control should receive focus first.
  • While open, the modal isolates sibling page branches with aria-hidden and inert so background content stays out of the active focus order.
  • If there is no focusable element inside the dialog, the modal container itself becomes the focus target.

Accordion

  • Use [data-fnlla-accordion] on the wrapper and [data-fnlla-accordion-button] on each trigger.
  • Each button should point to one panel with aria-controls, and each panel should point back with aria-labelledby.
  • Add data-fnlla-accordion-single only when the content model should allow one open item at a time.
  • Arrow Up, Arrow Down, Home and End move focus between triggers.

Toasts

  • Use [data-fnlla-toast] on the toast itself and [data-fnlla-toast-open] on any trigger that should reveal it.
  • Use [data-fnlla-toast-close] on dismiss controls inside the toast surface.
  • Optional data-fnlla-toast-autohide may be used in milliseconds for short-lived status confirmations.

Offcanvas

  • Use [data-fnlla-offcanvas-open] on the trigger and point it to one offcanvas selector such as #demo-offcanvas.
  • Mark the overlay with [data-fnlla-offcanvas] and keep one .offcanvas-panel surface inside it.
  • Use [data-fnlla-offcanvas-close] for every explicit dismiss control and keep a visible close button in the header or footer.
  • Use offcanvas for secondary workflows and side workspaces, not for critical blocking confirmation flows.

Popover and tooltip

  • Use data-fnlla-tooltip for short one-line or short-paragraph helper text attached to a trigger.
  • Use [data-fnlla-popover] with [data-fnlla-popover-toggle] when the panel needs richer content or small actions.
  • Popover panels should stay compact and should not become substitute modals or sidebars.

Scrollspy

  • Use [data-fnlla-scrollspy] on the wrapper and [data-fnlla-scrollspy-nav] on the navigation block.
  • Keep every scrollspy link pointing to a real section id on the same page.
  • Prefer scrollspy for long single-page references, internal guides and structured documentation pages.

Accessibility notes

Use these patterns consistently when wiring the components into a real project.

Status messages

  • Use role="status" for non-blocking confirmation and informational copy that should be announced politely.
  • Use role="alert" only when the message is urgent enough to interrupt assistive technology output.
  • Keep the copy brief and actionable so repeated announcements remain understandable.

Tables and dense data

  • Use real header cells and preserve reading order so assistive technology can move through rows and columns predictably.
  • If a table becomes too wide for the viewport, prefer a scroll container over visually collapsing columns into unreadable fragments.

Dropdowns

  • Use a real button for the toggle, not a plain link.
  • Pair the toggle with aria-expanded, aria-controls and a labeled menu container; the framework will wire aria-labelledby when needed.
  • Keyboard users should be able to open the menu, move through items with Arrow keys, Home and End, and close it with Escape.
  • When JavaScript is active, closed dropdown panels are also marked inert so focus cannot drift into collapsed content.

Tabs

  • Use real button elements for tabs so focus and keyboard behavior stay predictable.
  • The tab row should keep a clear visible label and uses a horizontal orientation by default.
  • If the project uses a vertical tab layout, declare aria-orientation="vertical" so Up and Down become the active arrow-key pattern.
  • Keep tab labels short enough to wrap cleanly on smaller screens, and test the stacked mobile layout.
  • Use tabs for sibling content sets, not for unrelated navigation destinations.

Navigation trails

  • Use breadcrumbs only when they reflect the real page hierarchy and help users move upward in that structure.
  • Mark the current breadcrumb with aria-current="page" and keep it non-interactive when it already represents the current page.
  • For pagination, make the active page obvious with both visible styling and aria-current="page".

Modals

  • Add role="dialog", aria-modal="true", aria-labelledby and aria-describedby.
  • Return focus to the trigger after closing where possible.
  • Use data-fnlla-modal-initial-focus when the first focus target should be explicit.
  • Keep background content out of the assistive-technology flow while the dialog is open.
  • Keep a visible close button in addition to backdrop and Escape support.

Offcanvas panels

  • Use dialog labeling just as you would for a modal when the panel carries task-level content.
  • Keep focus inside the open panel and return it to the trigger after close.
  • Reserve offcanvas for supplemental workflows that should not fully replace the current page context.

Tooltips and popovers

  • Tooltips should stay short, purely supportive and readable on focus as well as hover.
  • Popovers should include a clear close affordance when they contain more than static helper text.
  • Do not hide essential instructions only inside a tooltip if the task cannot be completed without them.

Accordions

  • Use buttons for each trigger and keep aria-expanded in sync.
  • Link each panel to its trigger with aria-labelledby.
  • Use a single-open pattern only when it matches the content model.
  • Support Arrow Up, Arrow Down, Home and End so keyboard users can move between headings quickly.

General interaction

  • Keep focus styles visible against every theme.
  • Respect prefers-reduced-motion for users who opt out of motion.
  • The framework ships a forced-colors fallback layer so core states stay legible in high-contrast system modes.
  • Test components with keyboard only, touch screens and narrow mobile layouts.
  • Use one shared navigation panel on mobile so primary links and CTA actions open and close together.
  • Use .visually-hidden when supporting text should stay available to assistive technology without remaining visible in the layout.
  • Prefer collapsed states that combine hidden, aria-hidden and inert so both focus order and assistive-technology output stay predictable.
  • Use toast notifications only for short, secondary feedback; anything blocking or actionable enough to demand focus should be a visible alert or dialog instead.

Scrollspy and section navigation

  • Use scrollspy only when the page has genuine long-form sections worth indexing.
  • Keep section headings meaningful enough that the side navigation reads as a real outline.
  • Prefer aria-current="location" for the active in-page destination.
