Choose your path

Most teams are either consuming the runtime in a downstream website or maintaining FNLLA UI itself. Starting on the right path cuts down a lot of unnecessary reading.

Consumer path

  • Start with team-usage-and-maintenance.html to understand the consumer-mode boundary.
  • Move to layout.html when the page shell or structure is still undecided.
  • Use component-classification.html before inventing a new reusable surface.
  • Open components.html, sections.html, forms.html or utilities.html only after the role is clear.

Maintainer path

  • Start with team-usage-and-maintenance.html to separate runtime work from downstream website usage.
  • Check api.html before changing hooks, selectors or supported runtime files.
  • Use distribution.html when publish, runtime export or handoff packaging may be affected.
  • Return to the reference pages only after the public contract and release boundary are understood.

Common tasks

Start from the task you are trying to solve, then jump into the deeper reference page only after the guide narrows the decision.

Starting a new website

Begin with team-usage-and-maintenance.html, then use distribution.html for packaging decisions and layout.html for the first page shell.

Choosing a reusable surface

Begin with component-classification.html, then open components.html, sections.html or forms.html once the category is clear.

Reviewing a markup change

Use the guides for intent, then verify the stable contract in api.html and the structural rules in layout.html.

Updating the framework itself

Start with team-usage-and-maintenance.html, then use api.html and distribution.html to check what is public, generated and release-sensitive.
