Which guide should you read first?

These guides sit above the raw component references. Use them when the question is not only what FNLLA Web ships, but how a team should decide, review or maintain that usage.

Choose your path

Most teams are either consuming the runtime in a downstream website or maintaining FNLLA Web itself. Starting on the right path cuts down a lot of unnecessary reading.

Consumer path

  • Start with team-usage-and-maintenance.html to understand the consumer-mode boundary.
  • Move to layout.html when the page shell or structure is still undecided.
  • Use component-classification.html before inventing a new reusable surface.
  • Open components.html, sections.html or utilities.html only after the role is clear.

Maintainer path

  • Start with team-usage-and-maintenance.html to separate runtime work from downstream website usage.
  • Check contract.html before changing hooks, selectors or supported runtime files.
  • Use contract.html when publish, runtime export, handoff packaging or public runtime hooks may be affected.
  • Return to the reference pages only after the public contract and release boundary are understood.

Common tasks

Start from the task you are trying to solve, then jump into the deeper reference page only after the guide narrows the decision.

Starting a new website

Begin with team-usage-and-maintenance.html, then use contract.html for handoff and public-boundary decisions and layout.html for the first page shell.

Choosing a reusable surface

Begin with component-classification.html, then open components.html or sections.html once the category is clear.

Reviewing a markup change

Use the guides for intent, then verify the stable contract in contract.html and the structural rules in layout.html.

Updating the framework itself

Start with team-usage-and-maintenance.html, then use contract.html to check what is public, generated and release-sensitive.
