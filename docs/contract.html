FNLLA Web contract and support boundary

This page combines the stable FNLLA Web packaging boundary and browser-facing contract: runtime files, handoff rules, documented class families, JavaScript hooks and helper methods. Maintainer-only checks and private repository details appear later and are not consumer dependencies.

Release state

Version

1.0.7 is the stable public release target for this repository snapshot.

Support model

Future changes should stay additive or conservative. Breaking markup, selector or hook changes should be treated as explicit upgrade work rather than routine drift.

Hardening baseline

Overlay behavior is now reconciled across one ordered modal or offcanvas layer stack, and fallback focus targeting skips CSS-hidden or fieldset-disabled controls before choosing the next usable control.

Public runtime contract

  • assets/css/fnlla-web.css is the supported runtime stylesheet for tokens, layout primitives, components, sections and utilities.
  • assets/js/fnlla-web.js is the supported runtime script for dropdowns, mobile navigation, tabs, accordions, modals, toasts, offcanvas panels, popovers, tooltips, cookie consent flows and scrollspy.
  • assets/icons/ is the supported local FNLLA Icons bundle for offline icon usage, including root-level SVG files, sprite.svg, the Lucide license file and the FNLLA notice.
  • The stable CSS API uses prefix-free class names such as wrapper, section, container, grid, btn, card and navbar.
  • contract.html defines how this runtime is packaged and when downstream projects should consume dist/fnlla-web/.
  • layout.html and the reference pages for components and sections define the documented authoring model that this contract expects.

Private implementation details

  • src/css/ and src/js/ are maintainer source layers, not consumer runtime dependencies.
  • scripts/fnlla-web-manifest.mjs coordinates publish and validation, but it is not part of the browser runtime API.
  • docs/assets/docs.css and docs/assets/docs.js are documentation-only assets and should not be linked from production pages.
  • Internal source-module ordering, source file names and docs helper selectors may change as long as the published runtime contract stays stable.

Choose the handoff

Distribution is mostly a decision about what another project should copy. The safe default is the generated runtime export, while the raw repository runtime files are mainly for framework maintainers working in-place.

Default: copy dist/fnlla-web/

Use the generated export when another project or another team needs FNLLA Web as a vendored dependency. It contains the runtime tree plus the minimal release files needed for version and license tracking.

dist/
  fnlla-web/
    assets/
      css/
        fnlla-web.css
      js/
        fnlla-web.js
      icons/
    LICENSE.md
    README.md
    VERSION

Use raw assets/ only when

  • You are working directly inside the FNLLA Web repository.
  • You need the live published runtime next to src/, scripts and docs while maintaining the framework.
  • You are validating that publish output, docs and browser behavior all describe the same current snapshot.

Runtime boundary

The supported runtime API is intentionally narrow. If a consumer page only depends on FNLLA Web, these are the files it should know about.

What belongs to runtime

  • assets/css/fnlla-web.css is the published stylesheet that pages should link.
  • assets/js/fnlla-web.js is the published script that pages should load for documented behaviors.
  • assets/icons/ holds the offline icon bundle, including root SVG files, sprite.svg, notice and license files.
  • dist/fnlla-web/ is the clean handoff form of the same runtime contract.
<link rel="stylesheet" href="fnlla-web/assets/css/fnlla-web.css">
<script src="fnlla-web/assets/js/fnlla-web.js"></script>

What stays outside runtime

  • Do not link src/css/... or src/js/.... Those are maintainer source files, not consumer dependencies.
  • Do not link docs/assets/docs.css or docs/assets/docs.js. They style and enhance the documentation product only.
  • Do not point production pages at docs-relative paths. Consumer pages should stay on runtime paths such as assets/css/fnlla-web.css and assets/icons/search.svg.

Repository zones

FNLLA Web has a source-to-runtime pipeline. Each zone has a distinct job and should stay on that side of the boundary.

Published runtime

assets/css/, assets/js/ and assets/icons/ are the stable browser-facing product surface inside the repository.

Runtime export

dist/fnlla-web/ is the downstream handoff package. It is generated, not hand-maintained.

Maintainer source

src/css/ and src/js/ are where maintainers change tokens, components and behavior modules before republishing readable runtime files.

Documentation product

docs/ proves and explains the framework contract, but docs/assets/ remains documentation-only support code.

Release discipline

Packaging stays trustworthy only when source, published runtime, generated docs and validation all move together.

Maintainer flow

  1. Edit source modules

    Change tokens, components or interaction code under src/ first.

  2. Republish runtime and docs

    Run node .\scripts\publish-fnlla-web.mjs so runtime files, generated guides and dist/fnlla-web/ all match source.

  3. Smoke-test browser behavior

    Run node .\scripts\test-fnlla-web-browser.mjs when runtime behavior or packaging changed.

  4. Validate the repository contract

    Run node .\scripts\validate-fnlla-web.mjs to confirm docs shell sync, release files and generated outputs.

Authoring baseline

The public API is not only files and hooks. It also includes the documented authoring model used to compose pages with stable layout responsibilities.

Recommended page shell

  • wrapper owns the full page shell.
  • Use semantic header, main and footer inside that shell.
  • section owns the full-width page band, spacing and optional background.
  • container, container-sm and container-lg own the inner content width.
  • grid and grid-2/3/4 own inner column layout.

Authoring rule

Start from the page shell and layout primitives first, then place content components such as card, panel, stat-card, forms or section-specific wrappers inside that structure. Cards are supported content components, but they are not the default page layout skeleton.

<div class="wrapper">
  <header>...</header>
  <main>
    <section class="section">
      <div class="container">
        <div class="grid grid-2 gap-md">...</div>
      </div>
    </section>
  </main>
  <footer>...</footer>
</div>

Document title contract

Professional browser-tab titles are supported through declarative HTML attributes and the shared runtime helper API.

Declarative HTML contract

Set title metadata on the root html element so the runtime can keep document.title consistent without page-specific scripts.

<html
  lang="en"
  data-fnlla-title-site="FNLLA"
  data-fnlla-title-page="Cookie Settings"
  data-fnlla-title-section="Docs"
>

Use data-fnlla-title-home="true" when the home page should collapse to the site name only.

Programmatic helper contract

Use window.FNLLAWEB.getDocumentTitle(), window.FNLLAWEB.getDocumentTitleConfig(), window.FNLLAWEB.syncDocumentTitle(config) and window.FNLLAWEB.setDocumentTitle(config) when dynamic routes, injected content or runtime state need to adjust the browser tab label.

The runtime composes a normalized Page | Section | Site title by default and de-duplicates repeated labels automatically.

Supported theme contract

Themes are applied through shared tokens, not through component-level rewrites.

Official theme values

default and dark are the currently documented built-in theme values.

Activation point

Apply data-fnlla-theme on body for production pages. Descendant theme wrappers are acceptable for documentation previews.

Customization rule

Framework maintainers should change shared tokens in src/css/tokens.css and republish assets/css/fnlla-web.css. Consumer projects that only use the published runtime should layer token overrides after assets/css/fnlla-web.css instead of editing component selectors first.

JavaScript hook contract

These hooks are the supported behavior API. The script assumes documented markup, real buttons for interactive triggers and valid target relationships.

Pattern Required hooks Expected target Notes
Dropdown data-fnlla-dropdown, data-fnlla-dropdown-toggle Toggle aria-controls should point to one .dropdown-menu Arrow keys, Home, End, outside click and Escape are supported.
Mobile navigation data-fnlla-nav-toggle aria-controls should point to one .navbar-panel The whole panel is the supported collapse target on mobile.
Tabs data-fnlla-tabs, data-fnlla-tab-list, data-fnlla-tab Each tab aria-controls should point to one .tab-panel The script normalizes role, selection state, focus order and panel visibility.
Accordion data-fnlla-accordion, data-fnlla-accordion-button Each trigger aria-controls should point to one .accordion-panel Add data-fnlla-accordion-single only when one-open-at-a-time behavior is desired.
Modal data-fnlla-modal, data-fnlla-modal-open, data-fnlla-modal-close data-fnlla-modal-open should point to a real selector such as #example-modal Focus trap, Escape, backdrop close and focus return are part of the supported behavior.
Toast data-fnlla-toast, data-fnlla-toast-open, data-fnlla-toast-close data-fnlla-toast-open should point to one real toast selector such as #save-toast Optional data-fnlla-toast-autohide uses milliseconds and is intended for non-blocking status feedback.
Offcanvas data-fnlla-offcanvas, data-fnlla-offcanvas-open, data-fnlla-offcanvas-close data-fnlla-offcanvas-open should point to one real offcanvas selector such as #workspace-panel Backdrop close, Escape close, focus return and layer isolation are part of the supported behavior.
Popover data-fnlla-popover, data-fnlla-popover-toggle, data-fnlla-popover-close The toggle aria-controls should point to one .popover-panel Outside click and Escape close are supported. Use this for compact rich helper content, not long-form dialogs.
Tooltip data-fnlla-tooltip The trigger itself is the supported target and should carry the tooltip text value Focus and hover both reveal the tooltip. Use optional data-fnlla-tooltip-position for top, right, bottom or left.
Cookie consent data-fnlla-consent, data-fnlla-consent-open, data-fnlla-consent-accept, data-fnlla-consent-save, data-fnlla-consent-category Optional settings modal should use one documented modal shell and each category toggle should carry a supported consent category value The runtime stores the state in one first-party consent cookie and dispatches a fnlla:consentchange document event when choices change.
Scrollspy data-fnlla-scrollspy, data-fnlla-scrollspy-nav Each nav link should point to one real section id on the same page The script applies aria-current="location" as sections become active in the viewport.
Range output data-fnlla-range-output, optional data-fnlla-range-prefix, optional data-fnlla-range-suffix The output value should match one documented .range-input[id] on the same page The runtime keeps the output text synchronized on load, input and change without page-specific JavaScript.
Runtime re-init window.FNLLAWEB.init(root) A document, element or injected subtree containing documented FNLLA Web markup The initializer is idempotent and is intended for progressive enhancement after dynamic DOM insertion.

Public runtime helpers

window.FNLLAWEB is the supported programmatic API. It is intended for progressive enhancement, dynamic content insertion and explicit open or close flows when markup alone is not enough.

Helper Accepted target Purpose
window.FNLLAWEB.init(root) A document, element or injected subtree Initializes documented FNLLA Web markup inside the provided scope. Calls are idempotent.
setTheme A theme name such as default or dark, with an optional selector string or direct element reference for a scoped wrapper Applies the selected documented theme to body by default or to one explicit theme wrapper.
getDocumentTitle, getDocumentTitleConfig, syncDocumentTitle, setDocumentTitle A config object or string page label when the title should be updated Reads or composes the supported browser-tab title from the FNLLA title metadata contract.
showModal, openModal, hideModal, closeModal A selector string such as #example-modal or the modal element itself Opens or closes a documented modal overlay.
showToast, hideToast A selector string such as #save-toast or the toast element itself Shows or hides a toast surface.
showOffcanvas, openOffcanvas, hideOffcanvas, closeOffcanvas A selector string such as #workspace-panel or the offcanvas element itself Opens or closes a documented offcanvas overlay.
openDropdown, closeDropdown A selector string pointing to the dropdown wrapper or the dropdown element itself Opens or closes a documented dropdown instance.
openPopover, closePopover A selector string pointing to the popover wrapper or the popover element itself Opens or closes a documented popover instance.
showTooltip, hideTooltip A selector string pointing to the tooltip trigger or the trigger element itself Shows or hides a documented tooltip.
refreshScrollspy A selector string pointing to the scrollspy wrapper or the wrapper element itself Refreshes active-link state for a documented scrollspy instance.
getConsentState, hasConsent, openConsentSettings, acceptConsent, rejectConsent, saveConsent, resetConsent An optional consent state object for save flows Reads or updates the documented first-party cookie-consent state without needing page-specific storage code.

Target resolution rule

Programmatic helpers accept either a selector string or a direct DOM element reference. If the target does not resolve to the documented FNLLA Web element type, the helper returns safely without throwing and keeps the chainable window.FNLLAWEB object contract.

The documented helper names are window.FNLLAWEB.setTheme(theme, target), window.FNLLAWEB.getDocumentTitle(), window.FNLLAWEB.getDocumentTitleConfig(), window.FNLLAWEB.syncDocumentTitle(config), window.FNLLAWEB.setDocumentTitle(config), window.FNLLAWEB.showModal(target), window.FNLLAWEB.openModal(target), window.FNLLAWEB.hideModal(target), window.FNLLAWEB.closeModal(target), window.FNLLAWEB.showToast(target), window.FNLLAWEB.hideToast(target), window.FNLLAWEB.showOffcanvas(target), window.FNLLAWEB.openOffcanvas(target), window.FNLLAWEB.hideOffcanvas(target), window.FNLLAWEB.closeOffcanvas(target), window.FNLLAWEB.openDropdown(target), window.FNLLAWEB.closeDropdown(target), window.FNLLAWEB.openPopover(target), window.FNLLAWEB.closePopover(target), window.FNLLAWEB.showTooltip(target), window.FNLLAWEB.hideTooltip(target), window.FNLLAWEB.refreshScrollspy(target), window.FNLLAWEB.getConsentState(), window.FNLLAWEB.hasConsent(category), window.FNLLAWEB.openConsentSettings(), window.FNLLAWEB.acceptConsent(), window.FNLLAWEB.rejectConsent(), window.FNLLAWEB.saveConsent(state) and window.FNLLAWEB.resetConsent().

Stable CSS families

The stable CSS API is grouped into reusable families. The items below are the supported categories to extend and reuse across projects.

Layout and tokens

.wrapper, semantic header, main and footer, .section, .section-alt, .section-surface, .section-primary-soft, .container*, .grid*, gap helpers and all shared --fnlla-* tokens. Use .section for the full-width band, .container for the inner page line, and .card* inside content areas rather than as the default page shell.

Core components

.btn*, .btn-group, .btn-toolbar, .close-btn, .card*, .panel*, .message*, .menu*, .chip*, .avatar*, .media*, .icon*, .alert*, .list-group*, .list-group-nav*, .empty-state*, .table*, .table-sticky, .table-status*, .breadcrumb*, .pagination*, .progress*, .progress-field, .progress-meta, .progress-label, .progress-value, .progress-indeterminate, .spinner*, .spinner-current, .loader-inline, .loading-*, .skeleton*, .skeleton-card, .skeleton-media, .skeleton-avatar, .skeleton-copy, .skeleton-actions, .skeleton-table, .skeleton-table-row, .search-* and the documented navigation patterns.

Forms and feedback

.form*, .fieldset*, .label, .input, .textarea, .select, .checkbox, .radio, .switch*, .range-*, .range-output, .range-labels, .file-*, .input-group*, .contact-form, .contact-field-*, .lead-capture-*, .form-layout-card, help text and success or error message blocks.

Interactive shells

.dropdown*, .tabs*, .accordion*, .modal*, .toast*, .offcanvas*, .popover*, .tooltip-panel and .scrollspy* are supported only with the documented hook contract.

Content primitives

Base content styling for blockquote, figure, figcaption, kbd, pre, code, description lists and the .prose wrapper is part of the stable authoring surface.

Section library

.hero*, .hero-copy, .hero-panel-intro, .hero-visual*, .hero-art-*, .hero-background*, .hero-inline-*, .feature-section*, .service-section*, .faq-section, .contact-section, .stats-section, .process-section, .pricing-section, .pricing-card-head, .testimonial-section, .cta-section, .cta-copy, .cta-inline-notes, .cta-proof-grid, .cta-support and .footer*. Utilities

Spacing, text alignment, weight, gap, display, flex, radius, overflow, opt-in scrollbar and accessibility helpers such as .scrollbar, .scrollbar-thin and .visually-hidden.

Maintainer checks and non-API boundaries

Stable does not mean frozen. It means changes should pass documented checks and preserve the supported authoring model for downstream projects.

Release checks for maintainers

Run the built-in publish, smoke-test and validation flow whenever runtime behavior, docs or packaging changed.

node .\scripts\validate-fnlla-web.mjs

When CSS or JavaScript source modules change, publish the readable runtime entrypoints with:

node .\scripts\publish-fnlla-web.mjs

That same publish flow now refreshes dist/fnlla-web/, the runtime-only handoff package for downstream projects.

Use the browser smoke test when runtime behavior or runtime publishing changes:

node .\scripts\test-fnlla-web-browser.mjs

Use the local browser matrix when you want the same smoke flow replayed across every detected supported local browser, including Firefox when available:

node .\scripts\test-fnlla-web-browser-matrix.mjs

The browser smoke coverage now includes mixed modal or offcanvas stacking, programmatic lower-layer close flows, fallback focus targeting for CSS-hidden or fieldset-disabled preferred targets, docs shell checks and runtime API behavior.

The repository also ships a GitHub Actions gate at .github/workflows/fnlla-web-hardening.yml so publish, validation and browser hardening can run automatically on GitHub before release-oriented merges.

Source-module ordering and the runtime-export boundary are centralized in scripts/fnlla-web-manifest.mjs so publish and validation read the same repository contract.

FNLLA Icons usage is intentionally static and local in this repository. The supported offline contract is local SVG files or the local sprite, not remote icon APIs or CDN-hosted assets. Source attribution to Lucide remains embedded in the local icon bundle.

What stays outside the stable API

  • Legacy hooks such as data-ui-target are not part of the supported API.
  • Legacy hooks such as data-ui-nav-menu are not part of the supported API.
  • Accordion behavior without a real aria-controls target is not part of the supported API.
  • Documentation-only selectors from docs/assets/docs.css should not be copied into product CSS dependencies.
  • Source-layer file paths inside src/ are not a consumer contract. Consumer projects should depend on published runtime assets instead.
  • Repository scripts and validation internals are not browser runtime dependencies even when they define how publish or validation work inside this repository.
